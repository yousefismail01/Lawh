# Phase 01.1: Madinah Mushaf Page View - Research

**Researched:** 2026-03-04
**Domain:** React Native Quran Mushaf layout, Arabic text rendering, page navigation
**Confidence:** HIGH

## Summary

This phase replaces the surah-list Home tab with a 604-page Madinah Mushaf view. The core challenge is rendering Arabic text in a fixed 15-line grid per page that visually matches the printed Madinah Mushaf, with RTL horizontal page navigation.

The recommended approach is **text-based rendering using the existing KFGQPCHafs Unicode font** with pre-computed line assignments from the Quran.com API v4. Each word in the API response includes `line_number` (1-15) and `page_number` (1-604), allowing exact reconstruction of the Madinah Mushaf layout. This data should be fetched once during a build/seed step and stored in local SQLite alongside the existing ayahs table. For page navigation, `react-native-pager-view` (v7, New Architecture / Fabric compatible) provides native RTL horizontal swiping with the `layoutDirection="rtl"` prop.

**Primary recommendation:** Use text-based rendering with the existing KFGQPCHafs font, pre-seeded word-level line assignments from quran.com API v4, and `react-native-pager-view` for RTL horizontal page swiping.

### Pattern 1: Word-Level Page Data Model
**What:** Store each Quran word with its page, line, and position so pages render from structured data, not dynamic text reflow.
**When to use:** Always -- this is the foundation of mushaf rendering.
**Example:**
```typescript
// New SQLite table: words
// Source: quran.com API v4 /verses/by_page/{page}?words=true
export const words = sqliteTable('words', {
  id: integer('id').primaryKey(),
  surahId: integer('surah_id').notNull(),
  ayahNumber: integer('ayah_number').notNull(),
  riwayah: text('riwayah').notNull().default('hafs'),
  position: integer('position').notNull(),     // word position within ayah
  pageNumber: integer('page_number').notNull(), // 1-604
  lineNumber: integer('line_number').notNull(), // 1-15
  textUthmani: text('text_uthmani').notNull(),  // Arabic word text
  charType: text('char_type').notNull(),        // 'word' | 'end' (verse marker)
})

// Index for fast page lookups
// CREATE INDEX idx_words_page ON words(page_number, line_number, position)
```

### Pattern 2: Page Rendering with Line Grouping
**What:** Query all words for a page, group by line_number, render each line as a justified row.
**When to use:** Core rendering pattern for every mushaf page.
**Example:**
```typescript
// In quranService.ts
async getWordsByPage(page: number, riwayah: Riwayah = DEFAULT_RIWAYAH) {
  return db.select().from(words)
    .where(and(eq(words.pageNumber, page), eq(words.riwayah, riwayah)))
    .orderBy(words.lineNumber, words.position)
}

// In useMushafPage hook -- group words into lines
function groupByLine(words: Word[]): Word[][] {
  const lines: Map<number, Word[]> = new Map()
  for (const word of words) {
    if (!lines.has(word.lineNumber)) lines.set(word.lineNumber, [])
    lines.get(word.lineNumber)!.push(word)
  }
  return Array.from(lines.values())
}
```

### Pattern 3: RTL PagerView Navigation
**What:** Use react-native-pager-view with `layoutDirection="rtl"` so swiping left goes to next page (higher page number), matching physical mushaf behavior.
**When to use:** Horizontal navigation mode (default).
**Example:**
```typescript

</PagerView>
```

### Pattern 4: Kashida (Tatweel) Justification
**What:** Insert tatweel characters (U+0640) between Arabic letters to stretch words to fill line width, simulating the printed mushaf's edge-to-edge justification.
**When to use:** Every text line in the mushaf (except centered lines like bismillah).
**Why not `textAlign: 'justify'`:** React Native's `textAlign: 'justify'` only adds inter-word spacing, does not insert kashida, and has known bugs with Arabic text (overflow issues on Android). For authentic mushaf appearance, manual kashida insertion or a custom justification approach is needed.
**Example:**
```typescript
// Approach: Render words in a flex row with justifyContent: 'space-between'
// For finer control, insert tatweel (U+0640) characters at eligible positions
// Eligible positions: before final-form letters, between connected letters
function MushafLine({ words }: { words: Word[] }) {
  return (
    <View style={{
      flexDirection: 'row-reverse', // RTL
      justifyContent: 'space-between',
      alignItems: 'baseline',
      paddingHorizontal: FRAME_PADDING,
      height: LINE_HEIGHT,
    }}>
      {words.map((word) => (
        <Text key={word.id} style={styles.word}>
          {word.textUthmani}
        </Text>
      ))}
    </View>
  )
}
```

### Anti-Patterns to Avoid
- **Global I18nManager.forceRTL:** Breaks non-Arabic UI elements. Use per-component `writingDirection: 'rtl'` and `flexDirection: 'row-reverse'` instead. (Established project pattern.)
- **Dynamic text reflow/line-breaking:** Never compute line breaks at runtime. The Madinah Mushaf has fixed line assignments per word -- use the pre-computed data.
- **Loading all 604 pages into memory:** Use lazy rendering. PagerView handles this natively; for vertical scroll mode use FlatList with windowSize.
- **QCF per-page fonts:** Requires 604 font files (~300MB). Stick with the single KFGQPCHafs Unicode font already loaded.
- **Using page images:** 604 images are huge (~150-300MB), not styleable for dark mode, and not accessible. Text rendering is the correct approach.

## Common Pitfalls

### Pitfall 1: PagerView with 604 Children
**What goes wrong:** Rendering 604 `<MushafPage>` components as direct children of PagerView causes massive initial load time and memory usage.
**Why it happens:** PagerView renders adjacent pages but React still instantiates all children.
**How to avoid:** Use PagerView's `offscreenPageLimit` prop (default 1) and wrap page content in lazy-loading wrappers that only render when the page is near-visible. Alternatively, use a state-based approach rendering only current +/- 1 pages.
**Warning signs:** App freezes on mount, high memory usage, slow initial render.

### Pitfall 2: Arabic Text Overflow with Tashkeel
**What goes wrong:** Arabic words with full tashkeel overflow their containers or get clipped, especially with `textAlign: 'justify'`.
**Why it happens:** Diacritics extend beyond the glyph bounding box; React Native's text measurement doesn't account for this perfectly. Known RN issue #32146.
**How to avoid:** Use the established `lineHeight = fontSize * 2` pattern (already in AyahText). For mushaf lines, use `flexDirection: 'row-reverse'` with `justifyContent: 'space-between'` instead of `textAlign: 'justify'`.
**Warning signs:** Text clipping at line edges, diacritics overlapping between lines.

### Pitfall 3: RTL FlatList Scrolling Bugs
**What goes wrong:** FlatList with `horizontal` and RTL has index/scroll position bugs on iOS, pages jump or scroll backwards.
**Why it happens:** Long-standing React Native bug with RTL + horizontal FlatList (issue #13100).
**How to avoid:** Use `react-native-pager-view` for horizontal mode (native implementation, no JS scroll bugs). Only use FlatList for vertical scroll mode (which doesn't have RTL scroll issues).
**Warning signs:** Pages display in wrong order, scrollToIndex jumps to wrong position.

### Pitfall 4: Zustand Persist Hydration Race
**What goes wrong:** App reads lastReadPage before AsyncStorage hydration completes, showing page 1 briefly then jumping.
**Why it happens:** AsyncStorage is async; zustand store initializes with defaults before persisted values load.
**How to avoid:** Use zustand persist's `onRehydrateStorage` callback or a loading state that waits for hydration before rendering the PagerView.
**Warning signs:** Flash of page 1 on every app open, then jump to actual last-read page.

### Pitfall 5: Surah Header/Bismillah Line Counting
**What goes wrong:** Pages with surah transitions render wrong number of lines because surah headers and bismillah occupy line slots.
**Why it happens:** The 15 lines per page includes surah headers and bismillah as "lines" in the mushaf. The quran.com API data accounts for this (line types include these), but naive rendering code may not.
**How to avoid:** The word-level data from the API already accounts for line assignments including headers and bismillah. Render the `line_type` field from the page data -- some lines are `surah_name`, some are `basmallah`, most are `ayah`.
**Warning signs:** Pages have 16+ visible rows, or text lines overlap with headers.

### Pitfall 6: Special Pages 1-2 Layout
**What goes wrong:** Pages 1 (Al-Fatiha) and 2 (beginning of Al-Baqarah) use centered layout with fewer text lines and more ornamentation, but code treats them as standard 15-line pages.
**Why it happens:** These pages are unique in the Madinah Mushaf and need special-case rendering.
**How to avoid:** Detect pages 1 and 2 explicitly and use a dedicated `MushafSpecialPage` component with centered text, extra vertical spacing, and additional ornamental elements.
**Warning signs:** Pages 1-2 look cramped or identical to regular pages.

## Code Examples

### Word Data Seeding Strategy
```typescript
// Seed word data from quran.com API v4
// Fetch once per page, store all words with line assignments
// Source: https://api.quran.com/api/v4/verses/by_page/{page}?words=true
async function seedWordsForPage(page: number) {
  const response = await fetch(
    `https://api.quran.com/api/v4/verses/by_page/${page}?language=en&words=true&per_page=50&word_fields=line_number,text_uthmani,page_number,position`
  )
  const data = await response.json()

  for (const verse of data.verses) {
    for (const word of verse.words) {
      // word.line_number: 1-15
      // word.page_number: 1-604
      // word.text_uthmani: Arabic text
      // word.char_type_name: 'word' | 'end'
      // word.position: position within verse
      await insertWord({
        surahId: verse.chapter_id,
        ayahNumber: verse.verse_number,
        position: word.position,
        pageNumber: word.page_number,
        lineNumber: word.line_number,
        textUthmani: word.text_uthmani,
        charType: word.char_type_name,
        riwayah: 'hafs',
      })
    }
  }
}
```

## Open Questions

1. **Kashida insertion quality**
   - What we know: `flexDirection: 'row-reverse'` + `justifyContent: 'space-between'` provides word-level spacing, but not character-level kashida stretching within words
   - What's unclear: Whether word-level spacing alone looks authentic enough, or if tatweel characters need to be inserted within words for true kashida appearance
   - Recommendation: Start with word-level flex spacing (simpler). If the visual result is not mushaf-authentic enough, add tatweel insertion as a refinement. The flex approach will work and look good; kashida within words is a polish item.

2. **Ornamental asset sourcing**
   - What we know: Need SVG assets for frame, surah banner, ayah markers
   - What's unclear: Whether to create custom SVGs, find open-source Islamic ornament assets, or use a design tool
   - Recommendation: Create simple, clean SVG assets. The frame can be a decorative border with corner flourishes. The surah banner is a rectangular header with arabesque edges. The ayah marker is a simple ornamental circle. Start minimal and refine.

3. **Word data seeding approach**
   - What we know: Quran.com API v4 provides all needed data; 604 API calls needed (one per page)
   - What's unclear: Whether to seed at build time (bundle JSON), at first-launch (API calls), or use QUL's downloadable SQLite
   - Recommendation: Bundle a pre-built JSON file (or SQLite) with the app. Fetching 604 pages from an API at first launch is slow and requires connectivity. The word data is static for Hafs riwayah -- bundle it.
