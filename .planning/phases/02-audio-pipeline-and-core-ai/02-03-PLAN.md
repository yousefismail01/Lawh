    1. Create stores/recordingStore.ts: Zustand store (NO persistence — recording state is ephemeral):
       ```typescript
       interface RecordingStore {
         status: RecordingStatus
         duration: number
         audioLevels: number[]  // last 50 levels for waveform
         filePath: string | null
         result: RecitationResult | null
         error: string | null

         startRecording: () => void
         updateDuration: (ms: number) => void
         pushAudioLevel: (level: number) => void
         stopRecording: (filePath: string) => void
         setUploading: (progress: number) => void
         setAnalyzing: () => void
         setResults: (result: RecitationResult) => void
         setError: (error: string, filePath?: string) => void
         reset: () => void
       }
       ```
       - audioLevels is a circular buffer of last 50 values (for waveform width)
       - State transitions enforce valid flow: idle -> recording -> uploading -> analyzing -> results/error

    2. Create lib/audio/offlineQueue.ts:
       - Uses expo-file-system to manage recordings/ directory
       - Each recording has: {id}.wav (audio) and {id}.meta.json (metadata sidecar)
       - `enqueue(filePath: string, riwayah: string): Promise<QueuedRecording>` — writes meta.json
       - `getPending(): Promise<QueuedRecording[]>` — reads all meta.json, filters status === 'pending'
       - `markUploading(id: string): Promise<void>` — updates status
       - `markUploaded(id: string): Promise<void>` — updates status
       - `markFailed(id: string): Promise<void>` — updates status (after 3 retries)
       - `retry(id: string): Promise<void>` — increments retryCount; if retryCount >= 3, calls markFailed instead
       - NEVER deletes WAV files — recordings are persistent
       - `initAutoUpload()`: Sets up NetInfo event listener. On connectivity restore (isConnected transitions false->true), calls getPending() and uploads each sequentially via inferenceService.submitRecitation. Uses 1s/2s/4s backoff between retries.
       - Show toast "Saved -- will analyze when online" when queueing offline (per user decision)

    3. Create hooks/useRecording.ts:

       - `startRecording()`: Request mic permission, call recordingService.start with audio level callback that updates store, start duration timer (setInterval updating every 100ms)
       - `stopRecording()`: Call recordingService.stop, enqueue to offlineQueue, if online call inferenceService.submitRecitation directly, else show offline toast
       - `retrySubmission()`: Re-submit from filePath in store
       - `clearResults()`: Reset store to idle
       - Clean up: clear interval on stop, handle component unmount
       - On interruption (expo-audio-stream fires stop event): auto-save to queue, update store status, show "Recording interrupted" banner. Offer to "Continue recording" (restart) or "Submit what was captured" (process the partial recording).

    4. Write offline queue tests using jest mocks for expo-file-system. Verify all behaviors listed above.
