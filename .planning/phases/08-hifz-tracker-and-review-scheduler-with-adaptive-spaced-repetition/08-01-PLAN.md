  <name>Task 1: SM-2+ type contracts and TDD algorithm</name>
  <files>lawh-mobile/lib/sr/types.ts, lawh-mobile/lib/sr/sm2plus.ts, lawh-mobile/lib/sr/sm2plus.test.ts</files>
  <behavior>
    - Test: Grade 0 (Again) resets repetitions to 0, interval to 1, consecutiveCorrect to 0
    - Test: Grade 2 (Hard) on first review sets interval to 1, increments repetitions
    - Test: Grade 3 (Good) on second review sets interval to 6, increments repetitions
    - Test: Grade 5 (Easy) on third+ review multiplies interval by ease factor
    - Test: Ease factor never goes below 1.3 even after many Hard grades
    - Test: Ease factor recovers (+0.05) after 3 consecutive correct (grade >= 3)
    - Test: Interval jitter produces values within +/-10% of computed interval (run 100 iterations, check bounds)
    - Test: Proportional overdue credit increases interval when card is overdue and grade >= 3
    - Test: Strength score is 0.0 for new cards (0 reps)
    - Test: Strength score approaches 1.0 as repetitions increase
    - Test: Strength score is always between 0.0 and 1.0
    - Test: Due date is computed correctly from today + interval days
  </behavior>
  <action>
    1. Create `lib/sr/types.ts` with shared type contracts:
       - `Grade = 0 | 2 | 3 | 5` (Again/Hard/Good/Easy per user decision -- NOT 0-5 range, only these 4 values)
       - `ReviewCard { easeFactor, interval, repetitions, dueDate, consecutiveCorrect }`
       - `SM2Result { easeFactor, interval, repetitions, dueDate, consecutiveCorrect, strengthScore }`
       - `HifzStatus = 'not_started' | 'in_progress' | 'memorized' | 'needs_review'`
       - `SurahStatus { surahId, totalAyahs, memorized, inProgress, needsReview, avgStrength }`
       - `ReviewQueueItem { surahId, ayahNumber, dueDate, intervalDays, easeFactor, repetitions, strengthScore, status }`

    2. RED: Write `lib/sr/sm2plus.test.ts` with all behavior tests above. Tests import from `./sm2plus`. Run tests -- all must fail.

    3. GREEN: Implement `lib/sr/sm2plus.ts` following the exact algorithm from RESEARCH.md Pattern 2:
       - `sm2plus(card: ReviewCard, grade: Grade, today?: Date): SM2Result`
       - `computeStrength(repetitions: number, easeFactor: number): number` (exported helper)
       - Wozniak EF formula: `EF + (0.1 - (5 - grade) * (0.08 + (5 - grade) * 0.02))`
       - Overdue credit: `Math.min(daysOverdue, interval) / interval` applied when grade >= 3
       - Recovery: +0.05 to EF when consecutiveCorrect >= 3 and EF < 2.5
       - Jitter: `1 + (Math.random() * 0.2 - 0.1)` multiplier
       - Min EF: 1.3
       - Strength: `min(1.0, (reps * EF) / (reps * EF + 8))`

    4. Run tests -- all must pass. For jitter test, use deterministic seed or test bounds over iterations.

    5. REFACTOR if needed: ensure function is pure (no side effects), well-commented.
  </action>

  <files>lawh-mobile/services/hifzService.ts, lawh-mobile/stores/hifzStore.ts</files>
  <action>
    1. Create `services/hifzService.ts` following `quranService.ts` patterns:
       - `openDatabaseSync('lawh-hifz.db')` with lazy init via `getDb()`
       - `CREATE TABLE IF NOT EXISTS hifz_progress` matching Supabase schema + `consecutive_correct INTEGER DEFAULT 0` and `mistake_count INTEGER DEFAULT 0` local columns
       - `CREATE TABLE IF NOT EXISTS review_schedule` matching Supabase schema
       - Indexes on `due_date` and `surah_id`
       - All queries include `AND riwayah = ?` (multi-riwayah ready)
       - Exported functions:
         - `initHifzDb()` -- create tables
         - `getSurahStatuses(riwayah: string): SurahStatus[]` -- aggregate query per surah
         - `getAyahProgress(surahId: number, riwayah: string): AyahProgressRow[]` -- per-surah ayah detail
         - `getReviewQueue(riwayah: string): ReviewQueueItem[]` -- due today or overdue, sorted by due_date ASC
         - `getReviewDueCount(riwayah: string): number` -- count only
         - `getTotalMemorized(riwayah: string): number` -- count of status='memorized'
         - `getJuzStats(riwayah: string)` -- strongest/weakest juz using juzBoundaries
         - `markAyahMemorized(surahId, ayahNumber, riwayah)` -- upsert hifz_progress with status='memorized', create initial review_schedule (interval 1 day, EF 2.5)
         - `markAyahInProgress(surahId, ayahNumber, riwayah)` -- upsert with status='in_progress'
         - `gradeAyah(surahId, ayahNumber, riwayah, grade: Grade)` -- read current card, compute sm2plus, write review_schedule + update hifz_progress strength/status/mistake_count/consecutive_correct
         - `getAyahCard(surahId, ayahNumber, riwayah): ReviewCard | null` -- for session use
       - Use `runSync` and `getAllSync` for synchronous operations (per-surah queries are small)

    2. Create `stores/hifzStore.ts` as a Zustand store (no persist middleware -- SQLite is the persistence):
       - State: `surahStatuses: SurahStatus[]`, `reviewDueCount: number`, `totalMemorized: number`, `loaded: boolean`, `sessionAyahs: ReviewQueueItem[]`
       - Actions:
         - `loadProgress(riwayah)` -- calls hifzService.getSurahStatuses + getReviewDueCount + getTotalMemorized, sets loaded=true
         - `gradeAyah(surahId, ayahNumber, riwayah, grade)` -- calls hifzService.gradeAyah, then refreshes affected surah status and review count
         - `markMemorized(surahId, ayahNumber, riwayah)` -- calls hifzService.markAyahMemorized, refreshes
         - `markInProgress(surahId, ayahNumber, riwayah)` -- calls hifzService.markAyahInProgress, refreshes
         - `loadReviewQueue(riwayah)` -- sets sessionAyahs from hifzService.getReviewQueue
       - Write-through pattern: every mutation writes to SQLite first, then updates Zustand state
  </action>
